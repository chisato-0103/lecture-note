# 保留ジョブ処理（中断に強い後処理）設計

## 背景・課題

現状の [`stopAndProcess`](../../../src/electron/main.ts) は「停止してノート化」を押すと、同一の非同期処理の中で
**①録音停止 → ②文字起こし（whisper large-v3-turbo） → ③要約（claude/ollama）** を一気に実行する。

②③は数分かかる重い処理で、この間に Mac のフタを閉じるとスリープして処理が中断・失敗する。
結果として `録音.wav` だけが残り、`文字起こし.txt` / `ノート.md` が生成されないフォルダが溜まっている。

大学内では接続 WiFi が頻繁に切り替わり、その間ネットが一時的に切れるため、
フタを閉じたまま処理を継続させても要約（ネット必須）が途中で止まる。

### 救いになる事実
- `録音.wav` は文字起こしの**前**（[`main.ts` の `recorder.stop()` 直後](../../../src/electron/main.ts)）に確実に保存される。
- **文字起こし（mlx_whisper）はローカル処理でネット不要**。ネットが必要なのは要約（claude）だけ。
- `文字起こし.txt` / `ノート.md` はいずれも `atomicWriteFile`（tmp→rename）で書かれるため、「ファイルが有る＝完成」と判定できる。

## ゴール

- 「停止してノート化」を押したら**すぐにフタを閉じて移動できる**（重い処理を待たない）。
- 中断（スリープ・WiFi 切替・アプリ終了・再起動）が起きても**成果を失わない**。
- 未完の録音は、次にアプリが動ける時（起動時・スリープ復帰時）に**自動で続きから処理**される。
- 手動操作を必要としない。

## 非ゴール（YAGNI）

- フタを閉じたまま処理を継続する仕組み（clamshell caffeinate）は作らない（発熱・電池・WiFi 切替のため不適）。
- 独立したジョブキュー DB は持たない（ファイルの有無を状態源とする）。
- 要約の途中再開（チャンク単位のレジューム）は作らない。要約は `文字起こし.txt` からやり直す。
- オフライン検知による要約スキップは作らない（失敗したら保留のまま次回再試行で足りる）。

## 設計

### 1. 状態はファイルの有無で表す

保存先ルート（`config.outputRoot`）配下の各録音フォルダ `YYYY-MM-DD_HHMM_録音/` の中身で処理段階を判定する。

| フォルダの状態 | 判定 | 実行する処理 |
|---|---|---|
| `録音.wav` あり / `文字起こし.txt` なし | 文字起こし未完 | 文字起こし（ローカル）→ 続けて要約 |
| `文字起こし.txt` あり / `ノート.md` なし | 要約未完 | 要約だけ（ネット必須）を再試行 |
| `ノート.md` あり | 完了 | スキップ |
| `録音.wav` なし | 対象外 | スキップ |

- 段階の区切りは `文字起こし.txt`（整形済み・cleaned）に置く。
  - 文字起こし段: `録音.wav` → whisper → 幻聴ループ除去（`removeHallucinationLoops`）→ `文字起こし.txt`。**すべてローカル・ネット不要**。
  - 要約段: `文字起こし.txt` → `summarizeTranscript` → `ノート.md`。**ネット必須（claude）または ollama**。
- whisper が副産物として出力する `録音.txt`（生テキスト）はフォルダに残るが、判定には使わない（判定は `文字起こし.txt`）。

### 2. ジョブメタデータ `job.json`

要約に必要な per-録音 の情報だけを、録音停止時にフォルダへ保存する。
語彙・エンジン・モデルは処理時点の現在設定を使うため保存しない（グローバル設定のため）。

```jsonc
{
  "recordedAt": "2026-07-09T10:30:00.000Z", // 並び順・表示用
  "materialPaths": ["/path/to/slides.pdf"]   // 添付資料（無ければ []）
}
```

- `job.json` が無いフォルダ（旧データや外部ファイル取り込み）は `materialPaths: []` 相当として扱い、要約は続行する。
- 要約時に `materialPaths` の各ファイルが移動・削除されていたら、その資料はスキップして要約を続行する（要約自体は止めない）。既存の `loadMaterials` が欠損に耐えること、または処理側で存在チェックしてから渡すことを実装で担保する。

### 3. 保留ジョブ処理モジュール（新規）

役割: 「保存先ルートを走査 → 未完フォルダを古い順に1件ずつ処理」。

- **多重実行ガード**: 処理中フラグ（メモリ上のブール／簡易ミューテックス）で、起動時トリガとスリープ復帰トリガが重なっても同時に走らないようにする。既に処理中なら「もう1周必要」フラグだけ立てて、現在の周回が終わったら再走査する。
- **走査対象**: `config.outputRoot` 直下のフォルダで、`録音.wav` があり `ノート.md` が無いもの。
- **除外**: 現在録音中のフォルダ（`activeRecordingDir`）はスキップする。
- **順序**: フォルダ名（＝タイムスタンプ）昇順で古いものから。
- **1件の処理**:
  1. `文字起こし.txt` が無ければ、文字起こし（`transcribe`）→ 整形（`removeHallucinationLoops`）→ `文字起こし.txt` を atomic 保存。
  2. 要約可否を判定（下記 4）。可なら `summarizeTranscript` → `ノート.md` を atomic 保存。
  3. 例外時はそのフォルダを保留のまま残し、通知して**次のフォルダへ進む**（全体は止めない）。
- **完了通知**: 1件完成ごとに通知する。ただしバックグラウンド走査では**ノートの自動オープンはしない**（復帰時に複数まとまって完成した際、何枚も開くのを防ぐ）。

### 4. クラウド要約の同意と背景実行

- バックグラウンドではモーダルダイアログを出せないため、**背景要約は `config.cloudConsent === true`（または `engine !== "claude"`）のときだけ**実行する。
- 未同意（claude かつ `cloudConsent` false）の場合は、文字起こしまでで止めてフォルダを保留のまま残す。同意後（設定 or 次の停止時の同意ダイアログ）に、次回トリガで要約が走る。

### 5. 「停止してノート化」の非ブロッキング化

[`stopAndProcess`](../../../src/electron/main.ts) を次のように変更する。

1. `recorder.stop()`（`録音.wav` 保存）— 現状どおり。
2. `stopLiveCaption()` / `live` フォルダ削除 — 現状どおり。
3. **同意確認** `ensureCloudConsent()` — ユーザーがその場にいるので一瞬。claude 初回のみダイアログ。
4. `job.json` を書き込む（`recordedAt`, `materialPaths`）。
5. **即 `setState("idle")`** に戻す（すぐフタを閉じて移動できる／次の録音も開始できる）。
6. 保留ジョブ処理を**キック**（`await` しない）。その場で待てば従来どおり完成する。

- 文字起こし・要約の本体は `stopAndProcess` から外し、保留ジョブ処理モジュールに一本化する（停止直後の処理も同じ経路を通す）。
- 状態機械: 背景処理は tray の `state`（idle/recording）とは独立に走る。背景処理中でも録音開始は可能とする（次の授業が優先）。CPU 競合の抑制（録音中は背景処理を始めない等）は「録音中フォルダをスキップ」+「1件ずつ逐次」で足りる範囲にとどめる。

### 6. トリガ配線

- `app.whenReady`（[`main.ts`](../../../src/electron/main.ts)）内、初期化後に一度キック。
- `powerMonitor.on("resume", ...)` を登録してキック。
- いずれも多重実行ガード経由で走査する。

## エラー処理・境界

- 走査対象が無ければ何もしない（正常）。
- 文字起こし失敗（無音・mlx_whisper エラー）: そのフォルダは保留のまま、通知して次へ。無音判定は既存の `MIN_TRANSCRIPT_CHARS` ガードに委ねる（要約段で例外→保留）。
- 要約失敗（ネット断・claude エラー）: `文字起こし.txt` は残るため、次回トリガで要約だけ再試行される。
- `job.json` 破損/欠損: `materialPaths: []` として扱い要約続行。
- 同一フォルダの二重処理: 処理中フラグ＋逐次処理で回避。

## テスト方針（Vitest・ロジック層優先）

- **段階判定**: フォルダ内のファイル構成（`録音.wav`/`文字起こし.txt`/`ノート.md` の有無）から「次にやること」を返す純関数のテスト（全組み合わせ）。
- **走査・順序**: 複数フォルダを与え、対象抽出・古い順ソート・録音中フォルダ除外が正しいか。
- **要約可否判定**: `engine` と `cloudConsent` の組み合わせで背景要約の可否が正しいか。
- **job.json**: 読み書き、欠損・破損時のフォールバック。
- **資料欠損**: `materialPaths` に存在しないパスが含まれてもスキップして続行すること。
- Electron の `powerMonitor`/`Tray`/`Notification` などの UI 副作用はテスト対象にしない。処理の中核はロジック関数に切り出してテスト可能にする。
