# 文字起こし精度改善 設計書

日付: 2026-07-17(レビュー反映済み)
対象: lecture-note(講義録音→文字起こし→要約ツール)

## 背景と問題

過去13件の録音データ(`~/Documents/講義ノート/`)を調査した結果、文字起こし精度の問題は3系統に整理できる。

1. **幻聴ループ**: 複数の録音で同一行の大量反復が発生している(「はい、どうぞ」99回連続、「鉄の原子力を使って、」99回連続、「奥…」115回連続など)。無音・低音質区間で Whisper が直前出力を引きずって反復する典型症状。原因は `mlx_whisper` 呼び出し(`src/pipeline/transcribe.ts:33-46`)が `condition_on_previous_text` を既定値(True)のまま使っていること。
2. **ループ除去の取りこぼし**: `removeHallucinationLoops`(`src/pipeline/clean.ts:23-48`)は「同一の正規化行が maxRepeats 回以上連続」で除去するが、反復ブロック中に空行が挟まると連続カウントがリセットされ、閾値未満の塊に分断されて残存する(実例: 「奥…」115回中5回が `文字起こし.txt` に残った)。実装時に生データで空行分断が実原因であることを再確認し、もし正規化で同一視できない近似行(行内反復等)が混在していれば normalize の強化を追加検討する。
3. **固有名詞の誤変換**: 「全寝付け職人」(→根付職人)、「福一」「豊田等」など。`vocabulary`(`--initial-prompt` に反映)が実運用で空のままで、語彙バイアスが一切効いていない。

## 変更内容

### 1. mlx_whisper デコードパラメータの追加(`src/pipeline/transcribe.ts`)

mlx-whisper 0.4.3 の CLI で実機確認済みのフラグを追加する:

- `--condition-on-previous-text False` — 直前ウィンドウの出力を次ウィンドウのプロンプトにしない。反復ループの根本抑制。副作用としてウィンドウ間の文体一貫性がやや下がる。`文字起こし.txt` 自体もユーザー可視の成果物のため、検証で正常録音の劣化有無を目視確認する(検証計画参照)。
- `--word-timestamps True` — 下記フラグの前提条件。処理時間が増える(検証で実測する)。
- `--hallucination-silence-threshold 2` — 幻聴が疑われる場合に2秒超の無音区間をスキップ。録音末尾の無音区間由来の幻聴に効く。

値はモジュール内定数とし、設定項目は増やさない(YAGNI)。

注意: `condition_on_previous_text=False` では `initial_prompt` は最初のウィンドウにのみ効く(True の場合でも長時間録音では2ウィンドウ目以降に直前出力へ置き換わっていくため、語彙バイアスは主に序盤に効くという実態は大きく変わらない)。

### 2. 幻聴ループ除去の空行バグ修正(`src/pipeline/clean.ts`)

`removeHallucinationLoops` の連続判定で空行を「透明」として扱う:

- 連続カウント中に空行が現れたら、カウントをリセットせず読み飛ばす
- **反復回数は空行を除いた実マッチ行数で数える**(読み飛ばした空行を回数に含めない)。既存テスト `clean.test.ts` の「空行は塊にしない」ケース(`"a\n\n\n\n\n\n\nb"` が変化しない)は維持されなければならない
- 除去が確定した反復ブロックは、ブロック**内部**(マッチ行とマッチ行の間)に挟まっていた空行も一緒に除去する。ブロック末尾側の空行(最後のマッチ行より後)はブロックに含めず、後続テキスト側に残す
- 先頭1行だけ残す挙動は現状維持。反復ブロック以外の行・空行はこれまで通り保持する

Vitest で「空行を挟んだ反復が除去される」「空行カウント水増しで誤除去されない(既存ケース)」「ブロック末尾に接する空行は残る」を追加・維持する。

### 3. 授業資料からの語彙自動抽出(新規 `src/pipeline/vocab.ts`)

授業資料が指定されているジョブでは、文字起こしの前に資料テキストから固有名詞・専門用語を抽出し、`config.vocabulary` と結合して `--initial-prompt` に渡す。

- **材料読込の前倒し**: 現状、CLI(`src/cli.ts`)も背景処理(`src/pipeline/pendingJobs.ts`)も資料読込は文字起こし後(要約段階)にある。両経路で `loadMaterials`(および pendingJobs では `readJobMeta`)を文字起こし前に移し、読み込んだ資料テキストは要約段階まで使い回す(PDF 再パースの二重実行をしない)
- 抽出: 要約と同じエンジンで行う。**既存の Summarizer 生成経路(makeSummarizer 相当)を再利用**し、claude 経路の API 課金ガード(`assertSafeClaudeAuth`)・`allowApiBilling`・タイムアウトの扱いを要約と同一にする。プロンプトは「固有名詞・専門用語を最大20個、カンマ区切りのみで出力」という短い指示。資料テキストは vocab.ts 側で先頭2万文字に切り詰めて送る(`loadMaterials` の6万文字上限とは別)
- 出力の検証: 返答をカンマ区切りでパースし、1語50文字超・空要素・重複を除去
- **initial_prompt の上限とトークン整合**(`buildInitialPrompt` の改修): Whisper はプロンプトを約223トークンに切り詰め、**末尾側を残す**。日本語は1文字1.5〜3トークンのため、現行の「600文字・手動語を先頭」では超過時に手動語から捨てられ優先順位が逆転する。次のように変更する:
  - 上限を全体150文字に引き下げ、超過時は**先頭側から**削る(末尾を残す)
  - 並び順を「自動抽出語 → 手動指定語(`config.vocabulary`)」とし、生き残り優先度の高い末尾に手動語を置く
- 同意ゲート(経路依存):
  - CLI: `--engine claude` の明示指定を同意とみなし抽出を実行(cloudConsent は参照しない。既存の CLI 要約と同じ扱い)
  - Electron 背景処理: 既存の `cloudConsent` が無ければ claude 抽出をスキップ(資料テキストは要約段階でも claude に送るため、同じ同意の範囲内)
- エラー時: 抽出に失敗しても文字起こしは止めない。警告ログを出して手動 vocabulary のみで続行する
- 資料が無いジョブでは何もしない(従来どおり手動 vocabulary のみ)

### 4. README 更新

デコードパラメータの変更と語彙自動抽出の挙動(資料があると自動で固有名詞ヒントが効くこと、claude 使用と同意の関係)を README に追記する。

## 検証計画

1. **ユニットテスト**(Vitest): clean.ts の空行ケース(上記3ケース)、vocab.ts のパース・切り詰め・失敗時フォールバック、buildInitialPrompt の末尾残し切り詰めと並び順、transcribe.ts の引数組み立て
2. **実測比較(パラメータ効果)**: 幻聴ループが顕著だった過去録音2件(`2026-07-09_1707`「鉄の原子力〜」99回、`2026-07-16_1443`「出席もっていますので」99回)+ 幻聴のない正常録音1件の `録音.wav` を新パラメータで再文字起こしする(出力はスクラッチ領域。既存データは変更しない)
   - パラメータ単体の効果は「旧・生 `録音.txt`」vs「新・生出力」で比較する(clean 適用前同士)
   - clean.ts 修正の効果は生出力に clean を適用した結果で別途確認する
   - 比較指標: 同一行の連続反復ブロック数と最大反復回数、処理時間(word_timestamps の影響)、正常録音での目視劣化チェック(脱落・文体破綻)
   - 成功条件: 生出力の最大連続反復が大幅減(目安 < 5)、正常区間のテキストが旧出力と同等以上、処理時間の増加が許容範囲(目安 2倍以内)
3. **語彙バイアスの効果確認**: 固有名詞誤変換が顕著だった `2026-06-25_1621`(根付職人)に対し、`ノート.md` 相当の資料テキストから抽出した語彙を initial_prompt に与えて再文字起こしし、「根付」等の対象語が正しく起こされるか(あり/なし比較)を確認する

## リスクと対応

- `word_timestamps=True` による処理時間増 → 実測して報告。許容できない場合は `--hallucination-silence-threshold` を外し `condition_on_previous_text=False` のみにフォールバック
- `condition_on_previous_text=False` による文体ゆれ・正常録音の劣化 → 正常録音1件の目視チェックで確認。問題があればパラメータ構成を再検討
- 語彙抽出の LLM 出力が不定形 → パース時の検証で防御。失敗してもパイプラインは止めない
- 資料由来テキストのプロンプトインジェクション → 資料はユーザー自身が選んだファイルで、既に要約段階で claude に送っている信頼境界の内側。抽出結果はパース検証と150文字上限で影響が限定される
